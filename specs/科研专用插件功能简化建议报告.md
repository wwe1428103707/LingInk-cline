# 科研专用插件功能简化建议报告

> 面向仅用于科研写作、文献管理、审稿与投稿辅助的产品形态  
> 目标：隐藏通用编程代理复杂度，突出科研任务入口

## 一、产品定位判断

当前 LingInk-cline 已集成 `academic-research-skills`，具备 deep-research、academic-paper、academic-paper-reviewer、academic-pipeline 等学术能力。但底层仍保留大量 Cline 通用 AI 编程代理入口，例如复杂模型供应商配置、Terminal、MCP、Plugins、Worktrees、Hooks、YOLO Mode、代码解释/修复等。

如果产品定位为“科研专用”，页面应从“能力平台”转为“科研工作台”。普通用户不需要理解底层 Agent、MCP、Provider、Terminal 或插件市场，只需要完成：

- 选题与研究设计
- 文献检索与综述
- 论文写作与润色
- 论证与引用检查
- 模拟审稿与返修
- 投稿格式与报告准备

核心原则是：底层能力保留，页面入口收敛；常规用户使用任务入口，进阶用户通过配置文件调整。

## 二、建议从页面中去掉或隐藏的功能

### 1. API Provider / 模型选择完整面板

当前设置页暴露大量 Provider：OpenRouter、Anthropic、OpenAI、Bedrock、Vertex、Ollama、LiteLLM、LM Studio、Groq、HuggingFace 等。

建议：

- 首次配置只保留一个“模型服务配置”入口。
- 默认推荐模型固定，不在主页面展示 provider 列表。
- 高级用户可通过配置文件或“高级设置”切换 provider、model、baseUrl、reasoning。
- `Use different models for Plan and Act modes` 默认关闭并隐藏。

理由：科研用户的心智模型是“功能驱动”，不是“模型驱动”。模型面板越复杂，越像开发工具，削弱灵砚的科研产品定位。

### 2. Terminal 设置页

当前有默认终端、Shell integration timeout、terminal reuse、execution mode 等。

建议：

- 从普通设置页移除 `Terminal` tab。
- 默认值固定：后台执行优先、终端复用开启、超时使用系统默认。
- 仅在配置文件中允许高级用户修改。

理由：科研工作流不应要求用户理解 VS Code terminal 细节。终端设置属于底层基础设施。

### 3. Worktrees / Git 并行任务

`Worktrees` 是面向软件开发的并行分支工作流。

建议：

- 首页和设置中隐藏 Worktrees。
- 相关按钮、当前分支提示、worktree modal 全部移除。
- 如果未来需要版本能力，应改成“论文版本/修订记录”，而不是 git worktree。

理由：科研用户需要“草稿版本、投稿版本、返修版本”，不是“代码分支”。

### 4. YOLO Mode / 自动执行无确认

当前 Features 中有 YOLO Mode，会自动执行任务并减少确认。

建议：

- 从页面移除 YOLO Mode。
- 默认关闭。
- 不建议通过 UI 开启，只允许配置文件启用。

理由：科研场景涉及引用、文件修改、投稿格式、数据结果，自动执行过强会带来信任风险。科研产品更重要的是可追溯、可确认、可解释。

### 5. Hooks 生命周期设置

Hooks 是高级扩展机制，面向开发者和自动化用户。

建议：

- 页面隐藏 Hooks。
- 保留底层能力给插件和进阶用户配置。

理由：用户只需要知道“灵砚会完成任务”，不需要管理生命周期钩子。

### 6. MCP Marketplace / MCP Display Mode / 远程服务器管理

MCP 对普通科研用户过于技术化。科研产品应该把 Zotero、Crossref、Semantic Scholar、arXiv、PubMed 等包装为“文献源”。

建议：

- Customize 中隐藏通用 MCP tab。
- 如果保留，只放到“高级工具连接”。
- `MCP Display Mode` 固定为 `rich` 或适合科研报告展示的默认值。
- 添加“文献源设置”替代 MCP 概念。

理由：用户要连接的是“Zotero / 学术数据库”，不是“MCP server”。

### 7. Plugins Marketplace

当前 Customize 有 Skills / MCP / Plugins。对科研专用版本，插件市场会分散注意力，也可能引入非科研能力。

建议：

- 普通页面隐藏 Plugins。
- 保留“学术技能包”安装状态。
- 进阶用户通过配置文件或开发者模式安装其他插件。

理由：科研产品应减少“可扩展性展示”，突出“开箱即用”。

### 8. 通用 Slash Commands

当前默认命令包括：

- `/newtask`
- `/deep-planning`
- `/smol`
- `/newrule`
- `/reportbug`

建议：

- 隐藏或替换默认 slash commands。
- `/reportbug` 放到帮助/关于页面。
- `/newrule` 仅进阶用户可见。
- `/smol` 改成系统自动压缩上下文，不暴露给用户。
- `/deep-planning` 改名为科研语义的“研究方案规划”。

理由：命令入口应该反映用户目标，而不是底层 Agent 行为。

### 9. 代码相关右键菜单

当前编辑器菜单有 Add to Cline、Explain with Cline、Improve with Cline、Fix with Cline，以及 Jupyter 代码单元相关命令。

建议：

- 对非代码文件隐藏 `Fix with Cline`、`Explain Code`、`Improve Code`。
- 对 Markdown、LaTeX、Word 转换文本保留科研语义菜单：
  - 润色选中文本
  - 检查论证
  - 补充引用
  - 生成回应审稿意见
- Jupyter 功能作为“数据分析辅助”可选模块，默认不突出。

理由：科研用户选中文本后期待的是论文处理，而不是代码处理。

## 三、建议保留但默认化的功能

| 功能 | 建议处理 | 理由 |
| --- | --- | --- |
| Auto Compact | 后台默认开启，不在主页面解释 | 长会话科研任务需要自动压缩上下文 |
| Checkpoints | 改名为“论文版本快照”，默认开启 | 符合论文修订心智 |
| Background Edit | 默认开启并隐藏 | 减少编辑器打断 |
| Telemetry | 保留在 General，文案改成中文科研产品隐私说明 | 涉及隐私透明 |
| History | 改名为“项目记录”或“会话记录” | 科研项目需要回溯 |
| Account | 涉及订阅/额度时保留，否则弱化 | 与商业化相关 |
| Academic Research Skills 安装 | 前置到首页或首次引导 | 学术能力是核心，而不是高级功能 |

## 四、页面中应该新增的快捷方式

### 1. 首页快捷任务区

建议首页直接展示 8 个主按钮：

- 选题引导
- 文献检索综述
- 论文结构重组
- 英文润色
- 漏洞扫描
- 模拟审稿
- 引用核查
- 投稿格式检查

理由：这些是科研用户最高频、最容易理解的任务入口，比“输入任意任务”更能降低使用门槛。

### 2. 选中文本右键快捷方式

建议新增：

- 润色选中文本
- 解释/界定核心概念
- 检查此段论证
- 查找支撑文献
- 生成反驳意见
- 改写为学术英文
- 提取摘要/关键词

理由：科研写作大量发生在局部段落层面，右键快捷入口比让用户写 prompt 更自然。

### 3. 论文项目快捷入口

首页或侧边栏新增：

- 新建论文项目
- 导入论文草稿
- 继续上次论文
- 查看项目记忆
- 查看待处理问题
- 导出报告

理由：科研任务不是一次性聊天，而是围绕一篇论文长期推进。需要从“任务历史”升级为“论文项目”。

### 4. 文献源快捷入口

新增“文献源”区：

- 连接 Zotero
- 导入 BibTeX / RIS
- 添加 PDF 文件夹
- 使用在线检索
- 查看文献库状态

理由：文献是科研产品的核心资产，应比 MCP、Provider 更靠前。

### 5. 审稿与投稿快捷入口

新增：

- 模拟审稿
- 生成 Response Letter
- 检查审稿意见回应是否充分
- 匹配目标期刊
- 生成 AI 使用声明
- 检查引用格式

理由：投稿前后是明确高价值场景，用户愿意为这些功能付出时间和费用。

### 6. 科研工作流模板入口

新增预设工作流：

- 开题准备
- 文献综述
- 初稿成文
- 投稿前检查
- 返修回应
- 会议报告准备

理由：相比“Marketplace / Workflows”，科研用户更容易理解“我现在处在哪个论文阶段”。

## 五、推荐的信息架构

建议主导航压缩为 5 个：

| 导航 | 内容 |
| --- | --- |
| 写作 | 选题、结构、润色、摘要、改写 |
| 文献 | 检索、综述、引用核查、Zotero |
| 审稿 | 漏洞扫描、模拟审稿、反驳演练、返修回应 |
| 项目 | 论文项目、记忆、历史、版本快照 |
| 设置 | 模型服务、语言、隐私、高级配置 |

“高级配置”折叠隐藏：

- Provider
- MCP
- Plugins
- Hooks
- Terminal
- Remote Config
- 调试工具

## 六、优先级建议

### 第一批：页面简化

- 隐藏 Terminal、Worktrees、Plugins、MCP Marketplace、YOLO、Hooks。
- 替换默认 slash commands。
- 首页增加科研快捷任务。
- 右键菜单改成科研语义。

### 第二批：产品化增强

- 项目首页。
- 文献源设置。
- 论文版本快照。
- Zotero / BibTeX / PDF 文件夹入口。

### 第三批：进阶能力开放

- 配置文件方式调整 Provider、MCP、Hooks、Terminal。
- 开发者模式显示插件市场和底层工具。

## 七、结论

灵砚不应该让科研用户看到 Cline 的“可编程 Agent 平台复杂度”。页面上应隐藏底层能力，保留科研任务入口，把默认值固定为安全、稳定、可追溯。这样产品会更像“科研写作工作台”，而不是“改皮后的编程助手”。
